1. Management samenvatting

Dit document beschrijft de gefaseerde aanpak voor het beschikbaar maken van vergaderruimtes (resources) in de Nextcloud Calendar. De implementatie verloopt in drie fasen:

Fase 1 — Vindbaar maken (MS365 Calendar Integration)

De bestaande MS365 Calendar Integration app maakt Exchange-resources vindbaar voor Nextcloud-gebruikers. Ruimtes verschijnen in zoekresultaten en hun beschikbaarheid (free/busy) is direct zichtbaar in de Calendar. Machtigingen zijn delegated — per gebruiker, alleen lezen.

↓

Fase 2 — Boekbaar maken (RoomVox)

RoomVox is een CalDAV-native room booking app die ruimtes als boekbare resources aanbiedt. Agenda's van Exchange-resources worden gesynchroniseerd zodat conflicten worden voorkomen. RoomVox gebruikt application permissions — een bewuste architectuurkeuze die de app onafhankelijk maakt van de Exchange-resourcestructuur.

↓

Fase 3 — Autonome werking (exit Exchange)

Door de Exchange-synchronisatie uit te schakelen wordt RoomVox de enige bron van waarheid voor ruimteboekingen. Dit is de exit-strategie: geen afhankelijkheid meer van Exchange.

2. Uitgangspunten en architectuurkeuzes

2.1 Twee apps, één doel

Het beschikbaar maken van resources in Nextcloud gebeurt met twee aparte Nextcloud-apps die elk een specifieke rol vervullen:

App Rol Permissiemodel
MS365 Calendar Integration Resources vindbaar maken (zoeken + free/busy) Delegated permissions (per gebruiker)
RoomVox Resources boekbaar maken (CalDAV room resources) Application permissions (server-niveau)

De scheiding is bewust: de MS365 Calendar Integration werkt namens individuele gebruikers en is daarmee beperkt tot wat die gebruikers zelf mogen zien. RoomVox opereert als server-applicatie en kan daardoor alle ruimte-agenda's beheren, ongeacht welke gebruiker een boeking maakt.

2.2 Application permissions vs. delegated permissions

RoomVox gebruikt application permissions (OAuth2 client credentials grant). Dit is een bewuste architectuurkeuze met de volgende overwegingen:

Aspect Delegated permissions Application permissions (RoomVox)
Authenticatie Elke gebruiker logt individueel in Eénmalige configuratie door beheerder
Scope Alleen agenda's die de gebruiker zelf mag zien Alle ruimte-agenda's van de organisatie
Afhankelijkheid Gebonden aan gebruikerssessies en persoonlijke tokens Onafhankelijk van individuele gebruikers
Achtergrondtaken Niet mogelijk (vereist actieve gebruikerssessie) Volledig ondersteund (cronjobs, webhooks)
Schaalbaar Token per gebruiker per ruimte Eén token voor alle ruimtes

Kernpunt: Bidirectionele agenda-synchronisatie vereist dat de server op elk moment — ook buiten gebruikerssessies om — kan lezen en schrijven op Exchange-agenda's. Delegated permissions ondersteunen dit niet, omdat ze afhankelijk zijn van een actieve gebruikerscontext. Application permissions zijn hiervoor de enige optie.

2.3 Onafhankelijkheid van Exchange resources

Een belangrijk architectuurprincipe is dat RoomVox niet afhankelijk is van de Exchange-resourcestructuur. De app synchroniseert uitsluitend agenda's (calendars), niet de Exchange-resourcedefinities zelf.

Dit betekent:

  • Ruimtes worden in RoomVox handmatig of via CSV-import aangemaakt met eigen metadata (naam, capaciteit, faciliteiten, locatie)
  • Per ruimte wordt optioneel een Exchange resource e-mailadres gekoppeld voor agenda-synchronisatie
  • De Exchange-koppeling is een synchronisatiekanaal, geen bron van de ruimtedefinitie
  • Als Exchange wegvalt, behoudt RoomVox alle ruimte-informatie en boekingen

Alleen agenda's worden gesynchroniseerd. Ruimtedefinities, capaciteiten, faciliteiten en andere metadata worden in RoomVox beheerd. Exchange is alleen de bron voor bestaande boekingen die buiten Nextcloud zijn gemaakt.

2.4 Exit-strategie

Door het ontwerp is de exit-strategie eenvoudig:

  1. Schakel de Exchange-synchronisatie uit (per ruimte of globaal)
  2. RoomVox wordt de enige bron van waarheid voor alle boekingen
  3. Bestaande boekingen blijven behouden in de CalDAV-agenda's
  4. Optioneel: verwijder de Azure AD app registration

Er is geen datamigratie nodig. Alle boekingen die via de synchronisatie zijn binnengekomen, staan al als reguliere CalDAV-events in de ruimte-agenda's van RoomVox.

3. Faseoverzicht

Fase Omschrijving App Vereist
1 Resources vindbaar maken in Nextcloud Calendar (zoeken + free/busy) MS365 Calendar Integration Entra ID App Registration (delegated permissions)
2 Resources boekbaar maken als CalDAV room resources met Exchange-synchronisatie RoomVox Entra ID App Registration (application permissions)
3 Autonome werking — Exchange-synchronisatie uitschakelen RoomVox Geen externe afhankelijkheden

Let op: Fase 1 en Fase 2 gebruiken aparte App Registrations in Microsoft Entra ID met verschillende permissiemodellen. Dit is een bewuste scheiding: de MS365 Calendar Integration werkt met delegated permissions (per gebruiker), RoomVox met application permissions (server-niveau).

4. Fase 1 — MS365 Calendar Integration

In deze fase wordt de MS365 Calendar Integration app geïnstalleerd. Hiermee worden Exchange-resources (vergaderruimtes) vindbaar voor Nextcloud-gebruikers.

4.1 Werking

De app registreert een virtueel adresboek en een Sabre DAV plugin in Nextcloud. Hierdoor kunnen gebruikers:

  • Zoeken naar resources — Exchange room resources verschijnen in de deelnemerslijst bij het aanmaken van een agenda-item. De app doorzoekt Microsoft Graph met meerdere strategieën (People API, Users API op naam/e-mail, fallback met prefix-matching).
  • Beschikbaarheid bekijken — Free/busy informatie van resources wordt realtime opgehaald via de Graph API (/me/calendar/getSchedule) en weergegeven in de Nextcloud Calendar planning view.

Beide functies werken transparant via het CalDAV-protocol, waardoor ze ook beschikbaar zijn in externe clients zoals Apple Calendar en Thunderbird.

4.2 Benodigde machtigingen

Machtiging Type Doel
User.ReadBasic.All Delegated Gebruikers en resources zoeken in de directory
People.Read Delegated Relevante contacten ophalen
Calendars.Read Delegated Free/busy informatie opvragen
Calendars.Read.Shared Delegated Free/busy voor gedeelde agenda's en resources
offline_access Delegated Refresh tokens voor langdurige sessies

Alleen leestoegang: De MS365 Calendar Integration vraagt uitsluitend read-machtigingen op. Er worden geen agenda-items, contacten of gebruikersgegevens gewijzigd in Microsoft 365.

4.3 Resultaat

Na Fase 1 kunnen Nextcloud-gebruikers Exchange-resources vinden en hun beschikbaarheid bekijken. Boeken is nog niet mogelijk — hiervoor is Fase 2 (RoomVox) nodig. De resources verschijnen als contacten in de deelnemerslijst, niet als CalDAV room resources.

5. Fase 2 — RoomVox

In deze fase wordt RoomVox geïnstalleerd. Dit is een CalDAV-native room booking app die ruimtes als boekbare resources beschikbaar maakt in Nextcloud Calendar en alle CalDAV-clients.

5.1 Architectuur

RoomVox maakt voor elke ruimte een verborgen Nextcloud-gebruikersaccount aan (prefix rb_). Dit account bezit een CalDAV-agenda en fungeert als principal voor het CalDAV scheduling-protocol. De accounts zijn onzichtbaar in zoekresultaten en login is altijd geblokkeerd.

De app registreert een Sabre DAV SchedulingPlugin die iTIP-berichten (REQUEST/CANCEL) onderschept voor ruimte-principals. Bij elke boeking wordt:

  1. Permissie gecontroleerd (Viewer / Booker / Manager rollen)
  2. Beschikbaarheid gevalideerd (dag/tijdregels, boekingshorizon)
  3. Conflicten gedetecteerd — zowel in de lokale CalDAV-agenda als in Exchange (als synchronisatie actief is)
  4. De boeking automatisch geaccepteerd of op tentative gezet
  5. De boeking naar Exchange gepusht (synchrone, fail-safe operatie)
  6. E-mailnotificatie verstuurd aan beheerders

Omdat ruimtes CalDAV room resources zijn, verschijnen ze automatisch in de ruimte-selector van Nextcloud Calendar, Apple Calendar en andere CalDAV-clients.

5.2 Exchange-synchronisatie

RoomVox synchroniseert uitsluitend agenda's (geen ruimtedefinities, geen metadata). De synchronisatie is bidirectioneel:

Richting Trigger Werking
RoomVox → Exchange Boeking via Nextcloud Direct (synchroon) gepusht naar Exchange via Graph API. Het event krijgt een RoomVoxSyncSource extended property om echolussen te voorkomen.
Exchange → RoomVox Webhook-notificatie of achtergrondtaak (elke 15 min) Delta-queries halen alleen gewijzigde events op. Nieuwe boekingen worden als CalDAV-events aangemaakt met X-ROOMVOX-SYNC-SOURCE: exchange.

Echopreventie

Om oneindige synchronisatielussen te voorkomen, gebruikt RoomVox een dubbel metadata-mechanisme:

  • iCal custom properties: X-ROOMVOX-SYNC-SOURCE (waarde: roomvox of exchange) en X-EXCHANGE-EVENT-ID in de CalDAV VEVENT
  • Exchange extended properties: RoomVoxSyncSource en RoomVoxUID op Exchange-events via de Graph API

Bij het pushen wordt een boeking met brontype exchange overgeslagen. Bij het pullen wordt een Exchange-event met brontype roomvox overgeslagen.

Webhooks

Voor near-realtime synchronisatie maakt RoomVox gebruik van Microsoft Graph webhooks. De webhook-subscriptions worden automatisch vernieuwd (maximale levensduur: 2 dagen) door een achtergrondtaak. Een periodieke synchronisatietaak (elke 15 minuten) fungeert als vangnet voor het geval webhooks mislukken.

5.3 Benodigde machtigingen

Machtiging Type Doel
Calendars.ReadWrite Application Lezen en schrijven van ruimte-agenda's voor bidirectionele synchronisatie
User.Read.All Application Valideren van resource e-mailadressen bij configuratie
Organization.Read.All Application Testen van de Exchange-verbinding

Belangrijk: Application permissions vereisen admin consent. Een Microsoft 365 Global Administrator moet de machtigingen goedkeuren. Deze machtigingen geven de applicatie toegang tot alle agenda's in de tenant. Het wordt aanbevolen om via Entra ID Application Access Policies de scope te beperken tot alleen de benodigde ruimte-resources.

5.4 Beveiliging

Aspect Implementatie
Client secret Versleuteld opgeslagen via Nextcloud IAppConfig (sensitive flag)
Token-caching Access tokens worden server-side gecachet met 60 seconden marge voor verlopen
Webhook-validatie Elke webhook-subscription heeft een 64-karakter cryptografisch willekeurig clientState dat bij elke notificatie gevalideerd wordt
Rate limiting Webhook-verwerking is rate-limited via distributed cache om thundering herd te voorkomen
SMTP-wachtwoorden Versleuteld opgeslagen via Nextcloud ICrypto
Graceful degradation Als Exchange niet bereikbaar is, blijft RoomVox volledig functioneel als standalone room booking systeem. Conflictcontrole valt terug op alleen de lokale agenda.

5.5 Resultaat

Na Fase 2 zijn ruimtes volledig boekbaar vanuit Nextcloud Calendar, Apple Calendar, Thunderbird en andere CalDAV-clients. Boekingen worden automatisch gesynchroniseerd met Exchange, zodat:

  • Boekingen in Nextcloud direct zichtbaar zijn in Exchange
  • Boekingen in Exchange (bijv. via Outlook) automatisch in Nextcloud verschijnen
  • Dubbele boekingen worden voorkomen door conflictcontrole op beide systemen

6. Fase 3 — Autonome werking

Fase 3 is de exit-strategie: het uitschakelen van de Exchange-synchronisatie, waardoor RoomVox de enige bron van waarheid wordt voor alle ruimteboekingen.

6.1 Stappen

  1. Schakel Exchange-synchronisatie uit — dit kan per ruimte of globaal via de admin-instellingen van RoomVox
  2. Verifieer bestaande boekingen — alle eerder gesynchroniseerde boekingen staan al als reguliere CalDAV-events in de ruimte-agenda's en blijven behouden
  3. Optioneel: verwijder de Azure AD app registration — de application permissions zijn niet langer nodig
  4. Optioneel: verwijder de Exchange-configuratie per ruimte (resource e-mailadres, sync-instellingen)

Geen datamigratie nodig. Omdat RoomVox ruimtedefinities en boekingen in eigen opslag beheert (Nextcloud IAppConfig en CalDAV-agenda's), gaat er niets verloren bij het uitschakelen van Exchange. De agenda's draaien gewoon door.

6.2 Resultaat

Na Fase 3 is RoomVox een volledig zelfstandig room booking systeem zonder externe afhankelijkheden:

Functie Status
Ruimtes beheren (CRUD, CSV-import/-export) Volledig operationeel
Boeken via Nextcloud Calendar Volledig operationeel
Boeken via Apple Calendar, Thunderbird, etc. Volledig operationeel (CalDAV)
Permissies (Viewer / Booker / Manager) Volledig operationeel
Beschikbaarheidsregels (dag/tijd, boekingshorizon) Volledig operationeel
E-mailnotificaties Volledig operationeel
Publieke API (v1) Volledig operationeel
Exchange-synchronisatie Uitgeschakeld

7. Vergelijking permissiemodellen

De twee apps gebruiken fundamenteel verschillende permissiemodellen. Dit is een bewuste ontwerpkeuze die voortvloeit uit de verschillende functionele vereisten:

Kenmerk MS365 Calendar Integration RoomVox
OAuth2-grant Authorization Code Flow + PKCE Client Credentials Grant
Permissietype Delegated Application
Authenticatie door Elke gebruiker individueel Beheerder, eenmalig
Leestoegang Alleen wat de gebruiker zelf mag zien Alle geconfigureerde ruimte-agenda's
Schrijftoegang Geen Ja (boekingen aanmaken/wijzigen/verwijderen)
Achtergrondverwerking Niet mogelijk Ja (cronjobs, webhooks)
Token-opslag Per gebruiker (ICrypto-versleuteld) Eenmalig, server-niveau (IAppConfig sensitive)
Admin consent Alleen voor User.ReadBasic.All Voor alle machtigingen

Waarom twee App Registrations? De MS365 Calendar Integration heeft alleen delegated (read-only) permissions nodig. Het toevoegen van application (read-write) permissions aan dezelfde registration zou het principe van least privilege schenden en de blast radius bij een eventueel security incident onnodig vergroten. Door de registraties te scheiden, blijft elke app beperkt tot precies de machtigingen die nodig zijn.

8. Risico's en mitigatie

Risico Impact Mitigatie
Application permissions bieden brede toegang Bij compromittering van het client secret kan een aanvaller alle agenda's in de tenant lezen/schrijven Gebruik Application Access Policies in Entra ID om de scope te beperken tot ruimte-resources. Roteer het client secret periodiek. Monitor sign-in logs.
Exchange-webhook mist events Boekingen gemaakt in Exchange verschijnen niet direct in RoomVox Achtergrondtaak (elke 15 minuten) fungeert als vangnet. Webhooks worden elke 24 uur automatisch vernieuwd.
Synchronisatielus (echo) Oneindige loop van push/pull operaties Dubbel metadata-mechanisme (iCal custom properties + Exchange extended properties) voorkomt echo's. Events worden bij de bron getagd.
Exchange onbereikbaar tijdens boeking Conflictcontrole onvolledig Graceful degradation: boeking wordt geaccepteerd op basis van alleen de lokale agenda. Synchronisatie haalt gemiste events later op.
Dubbele boeking in overgangsperiode Zelfde ruimte geboekt in zowel Exchange als Nextcloud voordat sync voltooid is Conflictcontrole bij boeking raadpleegt Exchange realtime. Webhook-latency is doorgaans < 1 minuut.
Client secret verloopt Synchronisatie stopt Monitoring via RoomVox admin panel (sync status). Aanbeveling: stel een herinnering in voor de verlooptermijn.

9. Performance tests

De RoomVox-codebase bevat een uitgebreide test suite van 176 tests met 308 assertions, waaronder specifieke performance tests die de schaalbaarheid van de booking engine valideren. Alle tests slagen op PHP 8.2 en 8.3 (CI/CD via Gitea Actions).

9.1 Testaanpak

De performance tests meten de PHP-logica overhead met alle I/O (database, netwerk) gemockt. Timing gebeurt met hrtime(true) voor nanoseconde-precisie. Dit isoleert de applicatielogica van externe latency, waardoor bottlenecks in de code zelf geïdentificeerd kunnen worden.

De testsuite dekt drie boekingspaden:

  • CalDAV/iTIP — het volledige scheduling-pad via het CalDAV-protocol (zoals Apple Calendar en Thunderbird)
  • Interne API — boekingen via de Nextcloud Calendar UI
  • Publieke API (v1) — boekingen via bearer-token API

9.3 Webhook-verwerking

Aparte performance tests valideren dat de webhook-handler snel genoeg reageert voor de 3-seconden deadline van Microsoft Graph:

Test Ruimtes Max inline sync Tijdslimiet
Webhook-verwerking 50 1 < 100 ms
Webhook-verwerking (stress) 300 1 < 500 ms
Webhook-verwerking (max inline 5) 300 5 < 500 ms

De webhook-handler verwerkt maximaal een instelbaar aantal ruimtes inline (synchroon) en plaatst de rest als achtergrondtaken in de queue. Rate limiting via distributed cache voorkomt overbelasting bij bulk-notificaties.

9.4 Productie-overwegingen

De performance tests meten uitsluitend PHP-logica. In productie komt daar I/O-latency bij:

Operatie Verwachte latency Toelichting
Database: conflictcontrole ~2–5 ms CalDAV calendarQuery op de ruimte-agenda
Database: boeking opslaan ~1–2 ms CalDAV createCalendarObject
Exchange: push ~50–200 ms HTTP-call naar Graph API (fail-safe, blokkeert geen respons)
Exchange: conflictcontrole ~50–200 ms HTTP-call naar Graph API calendarView

Totale verwachte doorlooptijd per boeking: circa 55–210 ms (PHP-logica + database). De Exchange-push is fail-safe: als Exchange onbereikbaar is, wordt de boeking alsnog lokaal geaccepteerd. De synchronisatie haalt het later in via de achtergrondtaak.
